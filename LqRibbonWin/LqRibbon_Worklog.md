# Codex Working Memory For KincoServo3

## 通用工程准则

- 细节优先。像边框、标题栏、按钮间距、hover/pressed、命中区域、拖拽释放状态、阴影这些“小细节”也属于功能质量，不能只看主流程能跑。
- 以真实效果为准。UI 类问题必须实际运行、截图或对照旧版本，不靠想象判断“差不多”。
- 少问多做。除非缺少关键信息会导致明显风险，否则优先用现有权限、源码、截图和构建结果推进。
- 不破坏用户现场。工作区可能有用户或历史改动；只修改任务相关文件，不还原、不格式化、不提交无关文件。
- 小步闭环。每次只解决一个明确问题：读代码、最小修改、编译、运行验证、清理产物、单独 commit。
- 代码可维护。重复坐标、颜色、文字和状态要抽常量或 helper；视觉位置变化时，事件命中区域也要同步变化。
- 跨平台意识。即使当前主要在 Windows 验证，也不要写死会破坏 Linux/macOS 的逻辑；平台相关代码要隔离。
- 编码安全。C++ 源码避免直接写中文，优先翻译资源或 UTF-8 转义，防止 MSVC C4819 和代码页问题。
- 验证要具体。最终反馈要说清楚编译了什么、运行了什么、截图/现象是什么、commit hash 是什么。
- 对用户指出的问题要直接修复根因，不要只做表面调整。例如标题栏 y 坐标不齐，应统一布局常量，而不是单独挪一个控件。

## UI/交互类问题的固定检查项

- 文本、图标、按钮是否在同一视觉基线或设计要求的位置。
- 控件是否贴边、重叠、超出、被遮挡，窗口最大化/还原后是否仍正确。
- 鼠标 hover、pressed、release、leave 后状态是否恢复。
- 拖动、调整大小、最大化拖出、贴边吸附、双击标题栏等系统手势是否符合预期。
- 视觉几何和点击命中区域是否一致。
- 动态语言切换、主题/字体/缩放变化后是否刷新布局和文案。
- 子窗口、弹窗、菜单、搜索结果等临时 UI 是否跟随主窗口状态正确显示和隐藏。

## KincoServo3 / LqRibbon 项目要求

- 架构固定为 Qt 5.15.2 C++。
- 代码严格遵循 `Qt代码规范.md`；LqRibbon 对外开源代码里的函数注释使用英文 Doxygen 风格。
- LqRibbon 源码放在 `LqRibbonWin/Code`，示例和测试入口放在 `LqRibbonWin/Example`。
- `LqRibbonWin` 只是区分 Python 版本的目录名，类名、命名空间和接口仍然叫 `LqRibbon`。
- LqRibbon 每完成一个小任务就单独 commit；只 commit LqRibbon 仓库，不 push。
- `KincoServoPro.pro` 项目除了 QtitanRibbon 替换成 LqRibbon 外，其他业务代码尽量不动。
- 替换后主项目中不应再出现 QtitanRibbon 依赖或接口痕迹；接口写法尽量保持 QtitanRibbon 兼容，只换库名。
- UI 目标是对齐旧 QtitanRibbon 版 Kinco Servo3 和最新版 Office 的 Ribbon 效果，不是重新设计。
- 需要兼容 Windows、Linux、macOS；Windows 无边框/原生边框手势要尽量完整。

## 已经踩过的具体坑

- 右上角最小化/最大化/关闭按钮容易重叠；几何计算必须统一使用常量，并同步 raise 和 hit-test 区域。
- 标题栏图标、文字、搜索框、窗口按钮不能分别写死 y 坐标；要用统一 caption top margin，否则会贴顶或错行。
- 搜索框只用 `tr("Search")` 不够；LqRibbon 作为库需要自己处理内置翻译，不能要求业务项目补 LqRibbon 翻译。
- MDI 子窗口不能保留 Windows 原生标题栏；拖动时不能出现释放鼠标后仍跟随的问题。
- Example 编译会生成 `.qmake.stash`、Makefile、release 目录，提交前必须清理。
- 主项目工作区有大量既有改动，不能误还原或误提交；LqRibbon 提交只 stage LqRibbon 相关文件。
- 自动化点击窗口坐标不稳定时，优先给 Example 加确定性 `--grab-preview` 预览参数。

## 默认执行流程

1. 先读相关代码、截图、旧实现和项目规范。
2. 明确最小任务边界，避免一次改太大。
3. 修改前确认仓库状态，保护用户改动。
4. 用统一常量/helper 修根因，不做只对当前截图有效的硬编码补丁。
5. 改 UI 时同步处理绘制、geometry、hit-test、状态、tooltip、语言切换。
6. 运行 `diff --check`，编译 Example。
7. 实际运行生成截图或启动主程序验证。
8. 编译主项目确认集成没有破坏。
9. 清理构建产物。
10. 单独 commit，最终回复说明验证结果和 commit hash。
