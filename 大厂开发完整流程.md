# 大厂开发完整流程：用 Codex + GitHub 跑通 LqRibbon

本文档给 `LqRibbon` 仓库使用，目标是把一个想法从需求、设计、开发、评审、合并、发布到复盘完整跑起来。流程参考大型研发团队常见做法，但保持足够轻量，适合个人项目、小团队和后续商业化协作。

当前仓库 GitHub 信息：

- 仓库：`LorenHan/LqRibbon`
- 默认分支：`main`
- 远程地址：`git@github.com:LorenHan/LqRibbon.git`
- GitHub CLI：已安装并登录，可直接使用 `gh`

## 0. 核心原则

大厂流程的本质不是“流程多”，而是让每一步都有可追踪、可复现、可回滚的证据链。

| 阶段 | 产物 | 工具 |
| --- | --- | --- |
| 需求进入 | Issue | GitHub Issues, `gh issue` |
| 方案澄清 | RFC / 设计说明 | Markdown, Codex |
| 开发实现 | 分支 + Commit | Git, Codex |
| 本地验证 | 测试结果 / 截图 / 日志 | pytest, Qt demo, Codex review |
| 代码评审 | PR Review | GitHub PR, Codex, `gh pr` |
| 合并发布 | Merge / Tag / Release | GitHub, GitHub Actions |
| 线上问题 | Hotfix / RCA | Issue, PR, Release |


检查当前仓库：

```bash
gh repo view --json nameWithOwner,defaultBranchRef,viewerPermission
```

设置当前目录默认仓库：

```bash
gh repo set-default LorenHan/LqRibbon
```

### 1.2 Codex

推荐从仓库根目录打开 Codex：

```bash
cd /Users/loren/Desktop/LqRibbon
```

使用 Codex 时，每个任务尽量说清四件事：

```text
目标：我要改什么？
上下文：涉及哪些目录、文件、报错、截图或 issue？
约束：不要改什么？要遵守什么风格？
完成标准：什么测试通过、什么现象消失、什么文档更新？
```

复杂任务先让 Codex 规划：

```text
先不要改代码。请阅读当前仓库结构，给我一个实现计划：
1. 需要改哪些文件
2. 风险点是什么
3. 要跑哪些验证
4. 你需要我确认什么
```

### 1.3 建议新增 AGENTS.md

`AGENTS.md` 是给 Codex 看的仓库规则。后续可以在仓库根目录添加：

```md
# AGENTS.md

## Repository Expectations

- 修改代码前先查看相关目录和现有风格。
- 不要无关重构。
- 修改 UI 后需要说明手动验证方式。
- 修改 Python 示例后运行相关 demo 或至少做 import 检查。
- 修改 C++/Qt 代码后说明构建环境和验证结果。
- PR 描述必须包含变更摘要、验证方式和风险。

## Review Guidelines

- 优先发现功能回归、跨平台问题、资源泄漏、UI 布局异常和缺失测试。
- 对文档拼写和格式问题可以低优先级指出。
```

## 2. 需求进入：先开 Issue

所有值得记录的工作都从 issue 开始。issue 是需求、bug、技术债和讨论的入口。

### 2.1 Bug issue 模板

```bash
gh issue create \
  --title "Bug: 描述问题" \
  --label bug \
  --body "## 现象

## 复现步骤
1.
2.
3.

## 期望结果

## 实际结果

## 环境
- OS:
- Python / Qt / Compiler:

## 附件
- 截图:
- 日志:
"
```

### 2.2 Feature issue 模板

```bash
gh issue create \
  --title "Feature: 描述能力" \
  --label enhancement \
  --body "## 背景

## 目标

## 非目标

## 用户故事
作为 ...
我希望 ...
以便 ...

## 验收标准
- [ ] 
- [ ] 

## 设计备注
"
```

### 2.3 Issue 分级

| 级别 | 含义 | 处理要求 |
| --- | --- | --- |
| P0 | 主分支不可用、数据破坏、发布阻断 | 立即修复，可走 hotfix |
| P1 | 核心功能严重异常 | 优先进入当前迭代 |
| P2 | 普通 bug 或明确体验问题 | 正常排期 |
| P3 | 优化、清理、文档增强 | 有空处理 |

## 3. 方案阶段：先把不确定性写下来

小改动可以直接开发。以下情况建议先写一个 RFC 或设计说明：

- 会改公共 API
- 会影响 `LqRibbonCPP` 和 `LqRibbonPy` 两套实现
- 会影响 UI 布局、资源加载、构建流程
- 需要引入新依赖
- 需要兼容旧行为

### 3.1 RFC 模板

```md
# RFC: 标题

## 背景

## 目标

## 非目标

## 当前实现

## 方案

## 备选方案

## 兼容性

## 风险

## 验证计划

## 发布计划
```

### 3.2 让 Codex 帮你补方案

```text
请基于 issue #123 和当前代码结构，帮我写一个轻量 RFC。
要求：
- 先描述现状
- 给出 2 个可选方案
- 推荐其中一个
- 列出风险和验证计划
- 不要修改代码
```

## 4. 开发阶段：一事一分支

### 4.1 更新主分支

```bash
git switch main
git pull --ff-only
```

### 4.2 创建分支

分支命名建议：

| 类型 | 命名 |
| --- | --- |
| 功能 | `feature/123-ribbon-tab-close` |
| Bug | `fix/123-titlebar-dpi` |
| 文档 | `docs/123-dev-workflow` |
| 重构 | `refactor/123-action-model` |
| 热修 | `hotfix/123-crash-on-startup` |

命令：

```bash
git switch -c docs/123-dev-workflow
```

### 4.3 给 Codex 的实现提示词

```text
目标：实现 issue #123。

上下文：
- 请先阅读相关 issue、README 和涉及目录。
- 重点查看 LqRibbonCPP/Code 和 LqRibbonPy/example。

约束：
- 保持现有风格。
- 不做无关重构。
- 不改未涉及的公共 API，除非先说明原因。
- 如果发现需求不清晰，先问我。

完成标准：
- 代码实现完成。
- 相关文档或示例同步更新。
- 运行可行的测试或验证命令。
- 最后给出变更摘要、验证结果和风险。
```

### 4.4 Codex 本地开发循环

推荐循环：

1. 让 Codex 读代码并给计划。
2. 确认计划后让 Codex 实现。
3. 让 Codex 自己跑测试、构建或 demo。
4. 打开 Codex review 面板检查 diff。
5. 对具体代码行留言，让 Codex 精修。
6. 让 Codex 做一次 `/review` 或“代码评审模式”。
7. 人工确认后提交。

常用提示：

```text
请 review 当前未提交 diff，重点看：
- 功能回归
- 边界条件
- UI 布局问题
- 跨平台问题
- 缺失测试或验证
只列问题，不要改代码。
```

```text
请根据刚才的 review 结果修复 P0/P1 问题，保持改动最小。
```

## 5. 本地验证：没有验证就不要开 Ready PR

### 5.1 通用检查

```bash
git status --short
git diff --stat
git diff
```

### 5.2 Python 侧建议

按实际依赖环境选择：

```bash
python -m compileall LqRibbonPy
```

如果 demo 环境可用：

```bash

```

### 5.3 C++/Qt 侧建议

按当前工程实际构建方式执行，例如：

```bash
cmake -S LqRibbonCPP -B build
cmake --build build
```

如果仓库当前还没有统一构建脚本，PR 中要明确写：

```text
未运行 C++ 构建：当前仓库未提供统一 CMake/Qt 构建入口。
已做静态检查：...
已做手动验证：...
```

### 5.4 验证记录格式

PR 里必须写清楚：

```md
## Verification

- [x] `python -m compileall LqRibbonPy`
- [ ] C++ build
  - 未运行原因：
- [x] Manual demo check
  - 环境：
  - 结果：
```

## 6. Commit：小步提交，信息可读

推荐 Conventional Commits：

| 类型 | 用途 |
| --- | --- |
| `feat` | 新功能 |
| `fix` | 修 bug |
| `docs` | 文档 |
| `refactor` | 不改变行为的重构 |
| `test` | 测试 |
| `chore` | 构建、脚本、维护 |

示例：

```bash
git add 大厂开发完整流程.md
git commit -m "docs: add Codex GitHub development workflow"
```

如果一个任务包含多个独立点，拆成多个 commit：

```text
fix: handle empty ribbon group title
test: add ribbon group title regression test
docs: document title behavior
```

## 8. Code Review：人审 + Codex 审

### 8.1 人类 reviewer 关注点

| 维度 | 检查问题 |
| --- | --- |
| 正确性 | 是否真正解决 issue？ |
| 兼容性 | 是否破坏旧 API、旧 UI、旧平台？ |
| 可维护性 | 命名、结构、边界是否清晰？ |
| 测试 | 是否覆盖核心路径和回归场景？ |
| 用户体验 | UI 是否稳定、清晰、无布局错位？ |
| 安全 | 是否引入不可信输入、路径、命令风险？ |

### 8.2 本地 Codex review

在 Codex 中可以直接说：

```text
请 review 当前分支相对 main 的全部变更。
按严重程度列出问题：
- P0: 必须修
- P1: 应该修
- P2: 可选优化
不要输出泛泛建议，只输出具体文件和原因。
```

### 8.3 GitHub 上触发 Codex review

如果仓库启用了 Codex cloud/code review，可以在 PR 评论：

```text
@codex review
```

也可以指定方向：

```text
@codex review for API compatibility and UI regressions
```

如果 Codex 给出 P1/P0，可以继续评论：

```text
@codex fix the P1 issue
```

### 8.4 用 gh 做 review

批准：

```bash
gh pr review --approve --body "LGTM"
```

请求修改：

```bash
gh pr review 123 --request-changes --body "请先补充验证记录。"
```

普通评论：

```bash
gh pr review 123 --comment --body "整体方向 OK，建议拆一下 commit。"
```

## 9. CI/CD：让机器守住底线

大厂流程里，合并前必须由机器做基础检查。

建议 GitHub Actions 至少包含：

- Python import / compile 检查
- C++/Qt 构建检查
- 单元测试
- 文档链接或格式检查
- 多平台矩阵，至少 macOS + Windows，如果项目目标包含跨平台

最小 Python 检查示例：

```yaml
name: CI

on:
  pull_request:
  push:
    branches: [main]

jobs:
  python-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: python -m compileall LqRibbonPy
```

合并规则建议：

- PR 至少 1 个 approval
- CI 必须通过
- 不允许直接 push 到 `main`
- 必须 squash merge 或 rebase merge，保持历史清晰
- 紧急 hotfix 可以简化 review，但不能跳过后续补充复盘

## 10. 合并：清晰历史，自动关闭 issue

合并前确认：

```bash
gh pr checks
gh pr view
```

PR body 中使用：

```text
Closes #123
```

合并后 GitHub 会自动关闭 issue。

推荐 squash merge：

```bash
gh pr merge --squash --delete-branch
```

本地同步：

```bash
git switch main
git pull --ff-only
git branch --delete docs/123-dev-workflow
```

## 12. Hotfix：线上问题快速修，但保留证据链

Hotfix 流程：

1. 创建 P0/P1 issue。
2. 从 `main` 拉 hotfix 分支。
3. Codex 协助定位和最小修复。
4. 本地验证。
5. 开 PR，说明为什么走 hotfix。
6. 至少一次快速 review。
7. 合并、打 patch tag。
8. 写 RCA。

命令：

```bash
git switch main
git pull --ff-only
git switch -c hotfix/123-crash-on-startup
```

RCA 模板：

```md
# RCA: 标题

## 时间线

## 影响范围

## 根因

## 修复

## 为什么测试没发现

## 后续行动

- [ ] 
```

## 14. 团队角色和职责

| 角色 | 职责 |
| --- | --- |
| Owner | 决定优先级、验收结果 |
| Author | 写代码、写测试、写 PR 描述 |
| Reviewer | 发现风险，不替 author 背锅 |

| CI | 自动执行稳定检查 |

Codex 适合做：

- 阅读陌生代码
- 生成实现计划
- 小步实现
- 补测试和文档
- Review diff
- 整理 issue/PR/release notes
- 定位 CI 失败

人必须负责：

- 需求判断
- API 取舍
- 兼容性承诺
- 发布决策
- 最终验收

## 16. 完整工作流示例

从一个 bug 到合并：

```bash
# 1. 建 issue
gh issue create --title "Bug: ribbon tab text clips on high DPI" --label bug

# 2. 创建分支
git switch main
git pull --ff-only
git switch -c fix/123-high-dpi-tab-text

# 3. 让 Codex 实现
# 在 Codex 中说明 issue、上下文、约束和完成标准。

# 4. 本地检查
git diff --stat
python -m compileall LqRibbonPy

# 5. 提交
git add .
git commit -m "fix: handle high DPI ribbon tab text"

# 6. 开 PR

# 7. 验证完成后 ready
gh pr ready

# 8. review 通过后合并
gh pr checks
gh pr merge --squash --delete-branch
```

## 18. Definition of Done

一个任务完成，必须满足：

- issue 或 PR 里能看懂为什么做
- 代码只解决当前问题
- 测试、构建或手动验证有记录
- PR 描述完整
- review 意见已处理
- CI 通过，或明确说明为什么没有 CI
- 合并后 issue 自动关闭或手动关闭
- 发布相关变更写入 release notes

这套流程跑顺后，Codex 的价值会越来越大：它不仅写代码，也能帮助保持工程纪律，让每次变更都更容易被理解、验证和回滚。
